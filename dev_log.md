### 3.19

- 解决同一 userId 同一 deviceId 的重复连接
    - 首先 redis 新值会把旧值顶掉，后端不会路由到旧连接上
    - 如果是合法断开重连，旧连接收到 FIN 后会做资源回收，如果是意外断开重连，心跳检测的收不到 pong，无法续期的旧连接也会做资源回收。回收本网关的
      map 记录，这里需要对比一下要回收的记录是不是旧连接，因为重连时完全有可能打到了同一网关上，新连接早就把记录顶掉了，旧连接不能动人家新连接的记录；其次回收
      redis 记录，很显然 redis 记录早被顶掉了，我们需要用连接 id（用雪花 id 正好合适）做 cas，这将必定失败
    - 如果是非法重连，也就是不断掉旧连接就开新连接的情况，旧连接在心跳检测后会为自己续期，同时续期 redis 记录，此时 cas
      会失败，立马断掉旧链接

- 把在线人数报告改为 bitmap 桶 + 滑动窗口
    - 在连接向 redis 注册或续期的时候顺便写入该时间段的 bitmap 桶中，如果是第一次写该桶则设置 3min 的
      ttl，无论何时后端都只拉取本分钟和上一分钟的桶做合并

- 改进 redis 的上下文，和 http 请求的 ctx 解耦
    - 对于注册和续期不能阻塞太久，以防 redis 网络抖动导致整个网关层请求堆积

### 3.25

- 完成 mq 双端的重试和确认

      挂了，本地打报错日志，不阻塞后续消息入库直接返回成功；如果消息无法路由，说明 go 节点挂了，本地打警告日志，也直接返回成功

### 3.27

- 双 token
    - 即 refresh token 和 access token，如果只用长 token，服务器无法再主动踢掉或封禁用户，只能等 token 过期，如果只用短
      token，每一次请求都要计算新的 jwt 或请求 redis，浪费性能
    - 前端拦截 401 的响应，再发一个刷新短 token 的请求，并且设置 isRefreshing 为 false，以防并发请求都来请求刷新，把后续请求异步，直到刷新的请求响应成功且业务
      code 为 200
- protocol buffer
    - 对于雪花 id，用 fixed64 绕开 varint 编码，对于 userId 和 senderId，用 varint 压缩，对于 content，用 bytes 绕过 utf-8
      校验，对于 type，用 enum，只在前端用数字

### 4.2

- 引入时钟轮算法
    - 为每个连接单独起一个 goroutine 去发心跳太奢侈了
    - 时间轮的每一个槽由链表和锁构成，每次轮询发心跳的时候把链表取出来，再把空链表放进去，拿着旧链表单独开一个 goroutine
      去执行，发完了再把链表放进新的时间槽里
    - 为连接增加 isClose 原子字段，如果连接断了就标记为 true，在发心跳轮询到该连接时，检查 isClose 为 true
      的话就从链表中踢出去，注意在连接关闭的时候，client 结构体是还在的，因为时钟轮的链表中还有它的引用，从链表中踢出时才被销毁
    - 时钟轮还要加个 tick 字段，用于记录上一次执行到的时钟，如果发生偏差就和当前真实时间做对比，进行追赶
- 为每个连接维护切片做缓冲区
    - 如果一个写动作卡住了，后续会堆积 mq 消息起的 goroutine
    - 还有一种方案是为每个连接维护一个写 goroutine 和 channel，但这样对内存不友好
    - 再增加一个 isWriting 原子字段，在把消息加入切片的时候检查有没有写 goroutine，没有就临时起一个，用完就销毁，依旧是拿切片然后放一个空切片，io
      操作不加锁
    - 用切片不用 channel 的好处是，不需要一个常驻的监听 channel 的 goroutine，而且不需要给缓存区大小可以动态调整
