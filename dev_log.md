## 开发日志：

### 小巧思
- 每个网关的 map 需要加读写锁，每个连接也需要加上锁
- redis 的路由记录用 string，在线记录用 zset，在连接量小的时候，可以直接对 string 结构 user:* 操作取路由，但在连接量大的时候，必须先用 userId 查在线记录，拿到对应设备拼接过后再查路由

### 3.19
- 解决同一 userId 同一 deviceId 的重复连接
    - 首先 redis 新值会把旧值顶掉，后端不会路由到旧连接上
    - 如果是合法断开重连，旧连接收到 FIN 后会做资源回收，如果是意外断开重连，心跳检测的收不到 pong，无法续期的旧连接也会做资源回收。回收本网关的 map 记录，这里需要对比一下要回收的记录是不是旧连接，因为重连时完全有可能打到了同一网关上，新连接早就把记录顶掉了，旧连接不能动人家新连接的记录；其次回收 redis 记录，很显然 redis 记录早被顶掉了，我们需要用连接 id（用雪花 id 正好合适）做 cas，这将必定失败
    - 如果是非法重连，也就是不断掉旧连接就开新连接的情况，旧连接在心跳检测后会为自己续期，同时续期 redis 记录，此时 cas 会失败，立马断掉旧链接

- 把在线人数报告改为 bitmap 桶 + 滑动窗口
    - 在连接向 redis 注册或续期的时候顺便写入该时间段的 bitmap 桶中，如果是第一次写该桶则设置 3min 的 ttl，无论何时后端都只拉取本分钟和上一分钟的桶做合并

- 改进 redis 的上下文，和 http 请求的 ctx 解耦
    - 对于注册和续期不能阻塞太久，以防 redis 网络抖动导致整个网关层请求堆积，对于删除收尾操作可以给更多的时间宽限

### 3.20
- 解决心跳风暴的问题
    - 对于强一致性要求的 redis 记录注册和注销，必须阻塞同步完成，对于弱一致性的续期，采用异步 Channel 收集 + Pipeline 批量执行，专门开一个 goroutine 去定时处理并根据 redis 返回值踢掉可能被顶掉的旧链接

### 3.21
- 解决 redis 重启导致的脚本丢失问题
    - 对于单条脚本，keepAliveScript.Run() 会自动把脚本交给 redis 缓存，并计算好哈希值方便后续指定使用哪个脚本，但是对于多条脚本的 pipe 发送，只会计算哈希而不管脚本的发送，需要自己处理，此外还要考虑 redis 重启时的脚本丢失。这里统一在续期失败后丢弃消息
