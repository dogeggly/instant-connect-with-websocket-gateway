### 4.2

- 引入时钟轮算法
    - 为每个连接单独起一个 goroutine 去发心跳太奢侈了
    - 时间轮的每一个槽由链表和锁构成，每次轮询发心跳的时候把链表取出来，再把空链表放进去，拿着旧链表单独开一个 goroutine
      去执行，发完了再把链表放进新的时间槽里
    - 为连接增加 isClose 原子字段，如果连接断了就标记为 true，在发心跳轮询到该连接时，检查 isClose 为 true
      的话就从链表中踢出去，注意在连接关闭的时候，client 结构体是还在的，因为时钟轮的链表中还有它的引用，从链表中踢出时才被销毁
    - 时钟轮还要加个 tick 字段，用于记录上一次执行到的时钟，如果发生偏差就和当前真实时间做对比，进行追赶
- 为每个连接维护切片做缓冲区
    - 如果一个写动作卡住了，后续会堆积 mq 消息起的 goroutine
    - 还有一种方案是为每个连接维护一个写 goroutine 和 channel，但这样对内存不友好
    - 再增加一个 isWriting 原子字段，在把消息加入切片的时候检查有没有写 goroutine，没有就临时起一个，用完就销毁，依旧是拿切片然后放一个空切片，io
      操作不加锁
    - 用切片不用 channel 的好处是，不需要一个常驻的监听 channel 的 goroutine，而且不需要给缓存区大小可以动态调整

# 4.16

- 为什么 seq_id 要用 redis 的 incr 生成
    - 雪花 id 没法去检测空洞，比如说先生成的雪花 id 的数据后落库，后生成的雪花 id 的数据先落库，然后在两个落库间前端来查离线消息了，就会永久跟新游标
    - 而 incr 可以让前端看到哪条消息没拉到，但这里必须保证 incr 的幂等自增，因为可能重试机制给两条同样的消息申请了两个
      seq_id，但只落库一个，前端就会迷惑
    - 为每个用户维护一个 hash，hashkey 是 msg_id，hashvalue 是 seq_id，再为每个键值对设置一小时的
      ttl，这样即使是数据申请了 seq_id 但没落库进入了死信队列，人工处理再入库时只要在一小时内就不会发生 seq_id 空洞，当然这么做
      redis 版本要 7.4 及以上，不然要去做两个哈希桶

# 4.17

- 梳理文件断点续传的相关问题
    - 引入上传时的状态机，0：未初始化，1：正在初始化，2：正在上传，3：完成
    - 对于初始化，先查数据库，有就直接返回，没有就用 redis cas。这里会有四种结局，set 成功，即 0 表示没有任务，这时状态已经设为了
      1，然后初始化任务，初始化完再改为 2。如果没 set 成功，就去查，1 表示有其他人正在初始化任务，直接返回让前端等 1 秒再来查，一会就到
      2 了；2 表示任务已经初始化好了，去 minIO 里查分片信息；3 表示任务已经完成了，但为什么数据库没查到呢，因为在查数据库和
      redis 的间隙才正好有其他人入库，这时直接返回结果就行
    - 不同状态下的 ttl 设置是不一样的，1 为 10 秒，过期说明向 minIO 初始化卡了，或者初始化完业务返回卡了，当然这时 minIO
      会多出一个废掉的 uploadId，但是没关系；2 为 1 天过期说明用户 1 天还没传完，直接作废，同时
      minIO 里也要设置清理掉 1 天还未合并的数据；3 为 1 分钟，这是数据库插入的时间
    - 对于合并，如果产生并发冲突，或者自己之前合并过但是那个线程后续挂了，总之再合并的时候发现有人已经合并好了，就捕获错误，当做无事发生，合并完去写数据库，然后把状态改为
      3
    - 对于合并前就挂了，或者合并成功但是入库前挂了的情况，只能依赖前端的重试，等到 1 天还没重试成功，那也没招了，白传
